Introduction

This page describes the initial phase of development of this website and the decisions made. I created this website to share some interesting or useful pieces of my knowledge with others.

Site language

At first, I was thinking about writing this website using a lightweight markup language like Markdown or AsciiDoc, which are commonly used nowadays, especially in source code repositories on GitHub or GitLab services which automatically render them to their output HTML form. Their syntax is however extremely limiting compared to pure HTML, at least in Markdown. You can use HTML inside them, but, in my opinion, mixing the two completely different languages / syntaxes looks and feels ugly and wrong. Oftentimes, when writing Markdown, I need to create a definition list or a numbered list that contains multiple elements, incl. a code block, and the default options are, to my knowledge, so limited I end up using <table> or <ol> for purely presentation reasons. At least, HTML syntax allows some tags, e.g. <li>, to be unterminated, which helps it to fit more seamlessly into Markdown.

The only reason against manual HTML is its handling of whitespace. It forces a not-so-pretty syntax for embedding code blocks inside <pre> where all whitespace (even indentation) matters. You can use CSS to trim all leading whitespace. There are also other places where whitespace between tags is undesirable and might cause an unwanted space in the rendered output.

Overall, I decided to write the website manually in HTML, preferably the semantic flavor, for all my content. I decided not to conform to the XML syntax because it only allows the &lt;, &gt;, &amp;, &quot;, and &apos; entity references to be used safely. As I don't like generic unicode characters outside ASCII in my source codes and the character references (such as &#169;) are not descriptive enough, this limitation is quite serious for me.

Maybe you expected to learn about my choice of a natural language in this section. As you may have noticed, and probably did, I'm not a native English speaker. However, as you may know, English is the lingua franca of technology, therefore, it has many positives to write in English. I remember one man criticizing me for not writing comments in my code, saying that code comments in imperfect English are way better than no comments at all. I hope this also scales up to this whole website; I hope this website will be helpful to someone. I might also consider using Grammarly to fix my errors and sound more natural.

Site generation

As writing all HTML files on this website completely manually and copy-pasting common parts of the files would be quite error-prone, I knew I needed some kind of generator or templating system. There are many options.

At first, I was thinking about a solution using basic bash scripts and classic UNIX tools. Then, I thought Python might be a better alternative. After trying these, I knew I needed something statically typed and preferably immutable. I considered C, C++, Rust and Haskell.

I was familiar with C and a bit with C++. With C, I would have to implement almost everything from scratch, and C++ might be tricky at times. I was totally unfamiliar with Rust and Haskell. I knew I had Rust on my to-do learning list. I've finally chosen Haskell as my generator language as it's less controversial than Rust and, unlike Rust, I will likely never need Haskell in my professional life, so it's not a problem if I develop some bad habits in writing it. Also, it has some cool features other languages lack (most of which I will probably never really understand).

On the other hand, Haskell seems like an overkill for such a project. And it has some features I don't like, such as the convention of using camelCase instead of snake_case in function names and PascalCase in filenames. It makes sense in combination with the convention of naming other things, however, so I'm okay with it. Also, it has the disadvantage of being too focused on functional programming and somewhat neglecting object-oriented features, compared to, e.g., Scala, which, in turn, has other problems.

Description list format

At first, I prefered a <dt> and the corresponding <dd> to be rendered on the same line, in a grid-like view. CSS properties display: grid; and grid-template-columns: max-content auto were perfect for that. I also included grid-column-start for its ability to correctly render a scenario where dt and dd elements were not in a one-to-one relationship. On small screens, however, I would probably need to set an override to collapse the grid into one column not to overflow to the right. As I don't want to maintain any screen-size-based rules, I decided not to use the grid at all.

There is, however, a problem with max-content as the first column may become too wide on small screens.
